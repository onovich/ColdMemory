# 当前项目问题检视

## 旧版 Demo 的主要问题

1. 单文件耦合严重
   - 剧情数据、状态机、Canvas 小游戏、UI 结构和样式全部堆叠在一个文件中。

2. 运行条件不成立
   - 原文件是 React JSX，并依赖 lucide-react，但仓库没有 HTML 入口、打包工具、依赖声明，也无法直接在浏览器中运行。

3. 无法直接部署到 GitHub Pages
   - Pages 需要静态资源入口或明确的构建产物，原项目两者都没有。

4. 状态更新存在结构性风险
   - 原 EVA 组件在 effect 中依赖 score，导致循环逻辑与 React 重新渲染强耦合，后续扩展时容易出现重复注册、时序错误和性能问题。

5. 设计表达不够清晰
   - 目前更像线性叙事 Demo，而不是完整的放置点击游戏。资源系统、长期成长和失败反馈仍然偏弱。

6. 不利于后续 Unity 移植
   - 若继续沿用 UI 与逻辑混写方式，后续迁移时很难抽离出可复用的状态机和数据结构。

## 本次重构后的改进

- 已改为静态站点结构，可直接部署到 GitHub Pages。
- 已拆分为数据层、逻辑层、表现层，后续便于继续扩展。
- EVA 小游戏从主界面逻辑中抽离，职责更清楚。
- 文档补齐，项目方向、问题与路线更容易协作。

## 仍需继续解决的问题

- 缺少持久化存档
- 缺少平衡性与数值曲线设计
- 缺少更多点击与放置系统之间的联动
- 缺少音效、动效和更强的惊悚氛围塑造

## 踩坑记录 / 技术经验 (Lessons Learned)

1. **CSS 动画与 DOM 频繁重绘的冲突 (SVG Rotation 动画重置重置坑点)**
   - **现象**：原版基于 React 的 Virtual DOM 机制，状态更新只会修改对应文本节点， SVG 图标不会被销毁，动画正常播放。重构为纯 JS (Vanilla JS) 模板字符串渲染后，每次数据变更（如解构时每 20ms 进度更新、自动巡航时每 200ms 里程更新）都会导致所在 DOM 区域被重新替换。
   - **坑点**：DOM 的高频重建会导致带有 `animation` 的元素（如旋转的 `Loader` 图标）不停被销毁并重新创建。由于新元素动画总是从 0 开始，导致图标在 20ms 刷新时看起来**完全静止**，在 200ms 刷新时则表现为**鬼畜般来回抽搐**。
   - **解决方案**：在保持轻量级静态架构（不引入复杂 DOM Diff 库）的前提下，巧用 CSS `animation-delay` 的负值特性解决：在渲染图标时，将当前的绝对毫秒数转化为 1 秒动画周期内的负延迟，即 `style="animation-delay: -${(Date.now() % 1000) / 1000}s"`。这让每次新创建的节点都能“快进”到正确的动画帧，在视觉上完美接断点，实现平滑旋转。通过计算绝对时间而非强依赖持久 DOM 节点，优雅绕过了更新频次带来的影响。

2. **解构按钮点击瞬间回到顶部（已完成后回底，但点击瞬时仍回顶）**
    - **现象**：虽然“解锁完成后滚到底部”已经实现，但点击 `[ 解构下一组记录 ]` 进入 `isDecoding` 的瞬间，文本区仍会跳到顶部。
    - **根因**：这一步会触发终端区域重绘，且解码面板显示导致滚动容器可视高度变化；浏览器会叠加焦点/滚动锚点行为，覆盖一次性 `scrollTop` 设置。
    - **修复**：
       - 在点击 `decode` 动作时直接执行终端到底部兜底（而非只在状态推导中处理）。
       - 对滚动恢复采用多时点稳定写入（立即、下一帧、短延时）而不是单次写入。
       - 在滚动容器上设置 `overflow-anchor: none`，避免浏览器自动锚点把视图拉回。
    - **防回归检查项**：
       - 点击解构按钮当帧，`nearBottom` 应保持 `true`。
       - 点击后 50~150ms，`nearBottom` 仍应为 `true`。

3. **功能面板插入叙事流导致布局耦合与视觉冲突**
    - **现象**：系统升级面板插在文本流中，和剧情节点、最近事件、解码按钮相互争抢空间，容易出现“文本与操作区压叠”问题。
    - **根因**：把高频变化的叙事流与功能管理流放在同一滚动上下文，导致结构耦合过高。
    - **修复**：
       - 升级功能改为页脚固定入口（独立视图）。
       - 剧情页仅保留非阻塞引导事件，不把升级列表强插入正文。
    - **防回归检查项**：
       - 终端页只承载“剧情 + 当前动作 + 最近事件”。
       - 升级列表仅在升级视图渲染。

4. **绝对定位浮层按钮与正文互相遮挡**
    - **现象**：解码按钮使用 `position: absolute` 固定在内容底部，随着文本增长会视觉压住最近事件区域。
    - **根因**：操作区脱离文档流，滚动内容和操作内容没有稳定边界。
    - **修复**：将解码区改为终端主视图下方的固定布局槽位（仍在同一视图，但不脱离流式布局边界）。
    - **防回归检查项**：
       - 最近事件最后一条在任何状态下都不应被解码按钮覆盖。
       - 解码态与非解码态切换时，滚动容器高度变化需配合滚动稳定策略验证。

5. **“本地已修复”与“线上未生效”的错位判断**
    - **现象**：本地 `4180` 行为正确，Pages 线上仍表现旧问题，容易误判为“修复无效”。
    - **根因**：修复尚未 push 或部署 run 仍在进行中，线上资源尚未更新到对应 commit。
    - **修复流程规范**：
       - 每次前端交互修复完成后，先确认提交 SHA，再确认 Deploy Pages run 对应 SHA、状态与结论。
       - 只在 run `completed + success` 后进行线上行为复测。
    - **防回归检查项**：
       - 复测结论必须标注环境（localhost / Pages）与 commit SHA。
       - 禁止在 `in_progress` 部署阶段给“线上已修复”结论。
