# 当前项目问题检视

## 旧版 Demo 的主要问题

1. 单文件耦合严重
   - 剧情数据、状态机、Canvas 小游戏、UI 结构和样式全部堆叠在一个文件中。

2. 运行条件不成立
   - 原文件是 React JSX，并依赖 lucide-react，但仓库没有 HTML 入口、打包工具、依赖声明，也无法直接在浏览器中运行。

3. 无法直接部署到 GitHub Pages
   - Pages 需要静态资源入口或明确的构建产物，原项目两者都没有。

4. 状态更新存在结构性风险
   - 原 EVA 组件在 effect 中依赖 score，导致循环逻辑与 React 重新渲染强耦合，后续扩展时容易出现重复注册、时序错误和性能问题。

5. 设计表达不够清晰
   - 目前更像线性叙事 Demo，而不是完整的放置点击游戏。资源系统、长期成长和失败反馈仍然偏弱。

6. 不利于后续 Unity 移植
   - 若继续沿用 UI 与逻辑混写方式，后续迁移时很难抽离出可复用的状态机和数据结构。

## 本次重构后的改进

- 已改为静态站点结构，可直接部署到 GitHub Pages。
- 已拆分为数据层、逻辑层、表现层，后续便于继续扩展。
- EVA 小游戏从主界面逻辑中抽离，职责更清楚。
- 文档补齐，项目方向、问题与路线更容易协作。

## 仍需继续解决的问题

- 缺少持久化存档
- 缺少平衡性与数值曲线设计
- 缺少更多点击与放置系统之间的联动
- 缺少音效、动效和更强的惊悚氛围塑造

## 踩坑记录 / 技术经验 (Lessons Learned)

1. **CSS 动画与 DOM 频繁重绘的冲突 (SVG Rotation 动画重置重置坑点)**
   - **现象**：原版基于 React 的 Virtual DOM 机制，状态更新只会修改对应文本节点， SVG 图标不会被销毁，动画正常播放。重构为纯 JS (Vanilla JS) 模板字符串渲染后，每次数据变更（如解构时每 20ms 进度更新、自动巡航时每 200ms 里程更新）都会导致所在 DOM 区域被重新替换。
   - **坑点**：DOM 的高频重建会导致带有 `animation` 的元素（如旋转的 `Loader` 图标）不停被销毁并重新创建。由于新元素动画总是从 0 开始，导致图标在 20ms 刷新时看起来**完全静止**，在 200ms 刷新时则表现为**鬼畜般来回抽搐**。
   - **解决方案**：在保持轻量级静态架构（不引入复杂 DOM Diff 库）的前提下，巧用 CSS `animation-delay` 的负值特性解决：在渲染图标时，将当前的绝对毫秒数转化为 1 秒动画周期内的负延迟，即 `style="animation-delay: -${(Date.now() % 1000) / 1000}s"`。这让每次新创建的节点都能“快进”到正确的动画帧，在视觉上完美接断点，实现平滑旋转。通过计算绝对时间而非强依赖持久 DOM 节点，优雅绕过了更新频次带来的影响。
